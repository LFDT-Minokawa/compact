# crates.io Publication Plan — `compact-runtime` + `compact-runtime-macros`

**Date:** 2026-06-02
**Task:** Prod-16
**Scope:** Plan the path from local-path-dep to crates.io for the Rust runtime crates
emitted-against by `compactc --rust`.

This is a planning note, not an action item. No `Cargo.toml` or source changes are
made by Prod-16 itself — the goal is to record the trade-offs, dependencies, and
checklist so the maintainer can pull the trigger when upstream is ready.

## 1. Current state

### Crates in this workspace

| Crate                    | Version    | Where it lives        | Notes                                           |
|--------------------------|------------|-----------------------|-------------------------------------------------|
| `compact-runtime`        | `0.16.100` | `runtime-rs/`         | Library facade for generated code.              |
| `compact-runtime-macros` | `0.16.100` | `runtime-rs-macros/`  | `proc-macro` companion; exports `witnesses!`.   |

Both crates are version-coupled to the TypeScript sibling
[`@midnight-ntwrk/compact-runtime` `0.16.100`](../../../runtime/package.json). The
Rust crate's `description` calls this out explicitly:

> "Native Rust runtime facade for code generated by `compactc --rust`. Mirrors
> `@midnight-ntwrk/compact-runtime`."

### Metadata already in place (Prod-4 D3)

`runtime-rs/Cargo.toml` is publish-ready in terms of metadata:

- `license = "Apache-2.0"`
- `repository`, `homepage`, `documentation` URLs set.
- `readme = "README.md"`.
- `keywords = ["compact", "midnight", "compactc", "zk", "blockchain"]`
- `categories = ["cryptography::cryptocurrencies", "compilers"]`
- `[package.metadata.docs.rs] all-features = true` + `--cfg docsrs`.

`runtime-rs-macros/Cargo.toml` has the minimum (`license`, `repository`,
`description`) but is **missing** `homepage`, `documentation`, `readme`,
`keywords`, `categories`. That gap closes during the pre-publish checklist
(see §3).

### Local path deps that stay local

18 generated fixture crates under `tests-e2e-rust/contracts/*` (`tiny`,
`zerocash`, `election`, plus the 15 feature fixtures) all use
`compact-runtime = { path = "../../../runtime-rs" }` and are marked
`publish = false`. They're CI-only and never go to crates.io.

The only path edge that crosses the publish boundary is the link from
`compact-runtime` → `compact-runtime-macros` inside `runtime-rs/Cargo.toml`:

```toml
compact-runtime-macros = { path = "../runtime-rs-macros" }
```

That needs to become `{ version = "0.16.100", path = "../runtime-rs-macros" }`
before publishing — see §3.

### External version pins

`flake.nix` line 223 pins `compactc` at `0.31.104`, mirrored in
`compiler/compiler-version.ss` (`(make-version 'compiler 0 31 104)`). Generated
contracts emit `compact_runtime::check_runtime_version!("0.16.100");` at the top
of each `lib.rs` — that string is the runtime crate's version, not the compiler's.

`Cargo.toml` workspace deps target the published `midnight-*` ledger crates
on crates.io:

```toml
midnight-base-crypto      = "1"
midnight-transient-crypto = "2"
midnight-onchain-state    = "3"
midnight-onchain-vm       = "3"
midnight-onchain-runtime  = "3"
midnight-coin-structure   = "2"
midnight-zswap            = "8"
midnight-serialize        = "1"
midnight-storage          = "2"
```

These are caret ranges (`"1"` ⇒ `>=1.0.0, <2.0.0`). They need to be pinned
tighter before publish — see §3.

## 2. Publication decision matrix

Three scenarios, evaluated against "what does the user experience the day after
we publish?"

### Option A — Publish now (private, before upstream)

Push `compact-runtime 0.16.100` to crates.io today from this branch.

| Pro                                                       | Con                                                      |
|-----------------------------------------------------------|----------------------------------------------------------|
| Reserves the crate name `compact-runtime`.                | No `compactc --rust` binary in any official release.     |
| Lets `docs.rs` start indexing.                            | `cargo add compact-runtime` succeeds but is unusable.    |
| Discoverable on crates.io search.                         | First-time users will hit a dead end.                    |
|                                                           | Forks the source-of-truth: branch ≠ upstream `main`.     |

**Verdict:** Defensible only as a name reservation. If the goal is name reservation,
publish a `0.0.0` placeholder, not `0.16.100`.

### Option B — Publish on first upstream release

Couple the crate publication to the next `compactc` release. The trigger is:
"toolchain `0.31.105` ships with `--rust` flag enabled → `compact-runtime` goes
to crates.io the same day."

| Pro                                                       | Con                                                      |
|-----------------------------------------------------------|----------------------------------------------------------|
| Clean signal — binary and library land together.          | Requires upstream merge to land first.                   |
| One announcement covers both.                             | Coordination dance between binary + crate release.       |
| `check_runtime_version!` macro is consistent day 1.       |                                                          |

**Verdict:** Good baseline. Requires the codegen-rust branch to merge into upstream
first. If we're confident the merge is imminent, this is the natural choice.

### Option C — Hold for upstream integration ✅ **Recommended**

Wait until the `codegen-rust` branch is merged into
`github.com/LFDT-Minokawa/compact` `main`, then publish from there. The first
release tagged on upstream `main` that bumps `compact-runtime` is the one that
goes to crates.io.

| Pro                                                       | Con                                                      |
|-----------------------------------------------------------|----------------------------------------------------------|
| Source of truth is upstream, not a fork branch.           | Slowest path to crates.io.                               |
| Avoids "Cargo.toml that doesn't work" for users.          | Requires PR landing first (Prod-17).                     |
| `check_runtime_version!` enforcement is end-to-end real.  |                                                          |
| Sets a precedent for future bumps (`X.Y.Z` everywhere).   |                                                          |

**Verdict:** Recommended. Codegen + runtime are tightly coupled —
`compact_runtime::check_runtime_version!("0.16.100");` is the first line of every
generated `lib.rs`. Publishing the runtime crate before `compactc --rust` is
buildable from a tagged upstream release leaves users without a working compiler
to pair with it. The published runtime would be a library nobody can drive.

### Recommendation summary

> Adopt Option C. Treat the upstream merge of the `codegen-rust` branch (Prod-17)
> as the gating event. The first upstream-tagged release whose runtime version
> matches the macro string emitted by `compactc --rust` is the first to publish.

## 3. Pre-publication checklist

Run in order. Each item should be a separate commit on a `runtime-rs-publish`
branch — not as part of `codegen-rust`.

### Crate-name availability

- [ ] `cargo search compact-runtime` → verify the name is unowned, or owned by
      the Midnight Foundation crates.io account.
- [ ] If unavailable: fall back to `midnight-compact-runtime`. Update `name = ...`
      in `runtime-rs/Cargo.toml` and the macro emitted by codegen
      (`compactc --rust`'s `check_runtime_version!` path resolution).
- [ ] Same for `compact-runtime-macros` (fallback: `midnight-compact-runtime-macros`).

### Macro crate publish posture

- [ ] Decide whether `compact-runtime-macros` is public API.
  - **Public**: needed because generated code uses `compact_runtime::witnesses!`
    re-exported from the macro crate. The current `[dev-dependencies]` edge from
    macros → `compact-runtime` is fine; cargo handles dev-only cycles.
  - **Private**: set `publish = false` and inline the proc-macros into
    `compact-runtime` via the `proc-macro` lib type — but that would force the
    runtime crate itself to be a proc-macro crate, which it can't be (it already
    has runtime code). **Public it is.**
- [ ] Add the missing metadata to `runtime-rs-macros/Cargo.toml`:
      `homepage`, `documentation`, `readme = "../runtime-rs/README.md"` (or its
      own short README), `keywords`, `categories`, `[package.metadata.docs.rs]`.

### Dependency hardening

- [ ] In `runtime-rs/Cargo.toml`, change the macros dep to:
      ```toml
      compact-runtime-macros = { version = "=0.16.100", path = "../runtime-rs-macros" }
      ```
      The `=` pin guarantees the lockstep version policy described in §4.
- [ ] In root `Cargo.toml`, tighten `[workspace.dependencies]` from caret to
      a specific minor — e.g. `midnight-base-crypto = "1.0"` instead of `"1"`.
      Match whatever line the TS runtime is built against per `flake.nix`
      (ledger-8.0.2 currently).
- [ ] No git deps. No `*` deps. (Already true today — re-verify on the publish
      branch.)
- [ ] `cargo tree -p compact-runtime --duplicates` → confirm no version skew.

### Verify packaging

- [ ] `cargo publish --dry-run -p compact-runtime-macros` from the workspace root.
- [ ] `cargo publish --dry-run -p compact-runtime` — must be run *after* macros
      is published, because the dry-run resolves the macros crate from crates.io,
      not from the path.
- [ ] Inspect the `.crate` tarball: `cargo package --list -p compact-runtime` —
      confirm no stray fixture / target / snapshot files leak in.
- [ ] `cargo doc --no-deps -p compact-runtime` locally; open the rendered HTML
      and confirm the top-level module doc and `check_runtime_version!` page
      render cleanly. (docs.rs reproduces this exactly.)

### Release engineering

- [ ] Tag the release in git: `runtime-rs-v0.16.100`. (Distinct from the
      toolchain's `v0.31.105` tag.)
- [ ] Decide on a release-notes location. The runtime crate shouldn't share
      `CHANGELOG.md` with the compiler — recommend adding
      `runtime-rs/CHANGELOG.md` separate from the root `CHANGELOG.md` and
      `CHANGELOG_compactc.md`.
- [ ] Add a `cargo publish` step (or manual runbook) to the release workflow
      so future bumps don't go stale.
- [ ] Document the publish order: **macros first, runtime second**. Crates.io
      rejects a runtime release if its `compact-runtime-macros` dep isn't already
      indexed.

## 4. Version coupling

Three artefacts must share the same `X.Y.Z`:

```
  ┌─────────────────────────────┐
  │  compactc (Scheme codegen)  │   emits the macro call below
  │  flake.nix v0.31.104        │
  └──────────────┬──────────────┘
                 │ generates
                 ▼
  ┌─────────────────────────────────────────────┐
  │  contract/src/lib.rs (generated):           │
  │  compact_runtime::check_runtime_version!(   │
  │      "0.16.100"                             │
  │  );                                         │
  └──────────────┬──────────────────────────────┘
                 │ pinned to
                 ▼
  ┌──────────────────────────────┐     ┌──────────────────────────────┐
  │  compact-runtime  0.16.100   │ ←── │  compact-runtime-macros      │
  │  (Rust crate, crates.io)     │  =  │  0.16.100                    │
  └──────────────┬───────────────┘     └──────────────────────────────┘
                 │ mirrors
                 ▼
  ┌────────────────────────────────────────┐
  │  @midnight-ntwrk/compact-runtime       │
  │  0.16.100  (npm)                       │
  └────────────────────────────────────────┘
```

Note that the compactc version (`0.31.104`) is **independent** — it has its own
release cadence. The macro string emitted by codegen is the *runtime* version,
not the compiler version.

### Bump policy going forward

| Change                                                                 | Bump            |
|------------------------------------------------------------------------|-----------------|
| Codegen-only fix; runtime API unchanged.                               | `0.16.101` patch |
| New runtime symbols generated code references (e.g. new stdlib helper). | `0.17.0`  minor |
| Incompatible runtime API change.                                       | `1.0.0`   major |
| TS runtime changes wire format only — Rust runtime unchanged.          | Skip Rust bump; let `compactc --rust`'s macro check fail loudly. |

For every Rust runtime bump, the macro string emitted by `compactc --rust` must
update the same release. The TS sibling tracks the same number out of policy
(historically the TS and Rust runtimes have a 1:1 wire-format mapping; if that
ever diverges we revisit this).

## 5. Long-term considerations

### Upstream `midnight-*` ledger crates

`compact-runtime` depends on 9 `midnight-*` crates that are workspace deps today:

```
midnight-base-crypto, midnight-transient-crypto, midnight-onchain-state,
midnight-onchain-vm, midnight-onchain-runtime, midnight-coin-structure,
midnight-zswap, midnight-serialize, midnight-storage
```

All 9 must be **published on crates.io** before `compact-runtime` can publish.
Crates.io rejects path-only or git-only deps. Pre-publish gate:

- [ ] `cargo search midnight-base-crypto` and confirm each of the 9 crates
      exists on crates.io at the version we pin.
- [ ] If any is unpublished — coordinate with the ledger team. **Do not** vendor
      them into the runtime crate as a workaround; that creates a maintenance
      fork.

### docs.rs build environment

`[package.metadata.docs.rs] all-features = true` is already set. docs.rs builds
on a Linux container with limited compile time and no network. Verify:

- [ ] No `[target.'cfg(...)'.dependencies]` Linux-incompatible blocks.
- [ ] All transitive `midnight-*` crates build on docs.rs (they should — they're
      pure Rust today).
- [ ] Build doesn't depend on `CHEZSCHEMELIBDIRS` or other Scheme env vars
      (it doesn't — the Rust crate has no Scheme dependency).

### Yanking policy

If a published version has a soundness or security bug:

1. **Yank** the affected versions on crates.io (`cargo yank --version X.Y.Z`).
   This blocks new resolutions but doesn't break existing `Cargo.lock` files.
2. **Bump** to the next patch (e.g. `0.16.101`) with the fix.
3. **Annotate** `runtime-rs/CHANGELOG.md` with the yank reason + advisory link.
4. **Notify** through the same channel as compactc security advisories.

A yank does **not** rewrite history — the macro check `check_runtime_version!`
in already-built contracts continues to work against the yanked version. Force
re-deploy is a separate operational concern.

### Maintenance ownership

The publish credential should live in a team account, not a personal account.
crates.io supports team-owned crates via GitHub team handles. Pre-publish:

- [ ] Create or identify the team (e.g. `@LFDT-Minokawa/runtime-maintainers`).
- [ ] After first publish: `cargo owner --add github:LFDT-Minokawa:runtime-maintainers compact-runtime`.
