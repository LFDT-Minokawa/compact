# Compact → Rust codegen — user guide

Audience: a Rust developer who has used Compact's TypeScript backend
(`@midnight-ntwrk/compact-runtime`) and wants to compile a `.compact`
contract to a Rust crate instead.

For codegen contributors (people changing the compiler or runtime),
see the three contributor READMEs linked at the bottom of this page.

## What is `compactc --rust`?

`compactc --rust` lowers a Compact contract to a Rust crate that
depends on the [`compact-runtime`](../runtime-rs) crate. Generated
Rust mirrors the TypeScript backend's shape one-for-one: the
contract becomes a `Contract<PS, W>` struct, the constructor becomes
`Contract::initial_state(ctx, …)`, each impure circuit becomes a
method returning `Result<CircuitResults<PS, R>, CompactError>`, and
public ledger fields are exposed through a `Ledger<'_>` view.
Witnesses are declared in a generated `Witnesses<PS>` trait that
the host program implements once.

The Rust path is byte-compatible with the TS path: the on-chain
`StateValue` produced after running `initial_state` and every circuit
is the same bytes as `@midnight-ntwrk/compact-runtime` would
produce. This is enforced by the byte-parity tests under
[`tests-e2e-rust`](../tests-e2e-rust). You can mix Rust contracts
with TS clients (and vice versa) on the same ledger without any
representation drift.

## Quick start

The walked-through example is the canonical `tiny.compact` contract:
a tiny key-value store that records a `Field` value plus the public
key of the actor who wrote it. The source is at
[`examples/tiny.compact`](../examples/tiny.compact).

### 1. Build `compactc`

`compactc` is the Compact compiler. Build it from this repository:

```bash
# From the repo root, enter the dev shell that bundles compactc + Rust.
nix develop

# Verify compactc is on PATH and supports --rust.
compactc --help | grep -- --rust
# →   --rust causes the compiler to additionally emit a Rust crate (contract/lib.rs)
```

If you don't use Nix, build the compiler with Chez Scheme as
described in [`compiler/README-rust-passes.md`](../compiler/README-rust-passes.md).
The TL;DR is `nix build .#compactc` and then `./result/bin/compactc`.

### 2. Compile your contract with `--rust`

```bash
mkdir -p /tmp/tiny-out
compactc --rust --skip-zk --skip-ts \
  examples/tiny.compact /tmp/tiny-out/

ls /tmp/tiny-out/contract/
# → lib.rs   Cargo.toml (depending on emit settings — see note below)
```

Flag notes:
- `--rust` — emit the Rust crate. Without this flag you get only
  the TypeScript output.
- `--skip-zk` — skip proving-key generation. Useful for fast iteration;
  generated Rust does not depend on the proving keys.
- `--skip-ts` — skip emitting the TypeScript contract. Optional, but
  if you only care about Rust this avoids producing files you'll
  ignore. Must be combined with `--rust` so the compiler still has
  a contract-code target.

The emitted `contract/lib.rs` is a self-contained Rust source file.
It is annotated `Generated by compactc. Do not edit by hand.` —
treat it like a build artefact, not source.

### 3. Wire the generated crate into your project

The simplest layout: copy the generated `lib.rs` into your crate
under `src/contract/mod.rs` (or wherever you like), then depend on
`compact-runtime`. Minimum `Cargo.toml` dependencies:

```toml
[dependencies]
compact-runtime = { git = "https://github.com/LFDT-Minokawa/compact", branch = "main" }
# Or, once published to crates.io:
# compact-runtime = "0.16"
```

The generated `lib.rs` opens with:

```rust
use compact_runtime::*;
use std::marker::PhantomData;

compact_runtime::check_runtime_version!("0.16.100");
```

The `check_runtime_version!` macro is a compile-time assertion: if
the `compact-runtime` crate your contract is linking against has a
different version than the compactc version that produced the file,
the build fails with `compact-runtime version mismatch`. See
[Versioning](#versioning) below.

### 4. Implement `Witnesses<PS>` for your private state

Every Compact `witness foo(...): T;` declaration becomes one method
in a generated `Witnesses<PS>` trait. The trait method name is the
Compact name **with all non-alphanumeric characters replaced by `_`**
(so Compact `private$secret_key` becomes Rust `private_secret_key`).

For `tiny.compact`, the trait is:

```rust
pub trait Witnesses<PS> {
    fn private_secret_key<'a>(
        &self,
        ctx: &WitnessContext<Ledger<'a>, PS>,
    ) -> (PS, [u8; 32]);
}
```

Each method returns `(updated_private_state, witness_value)`. The
host implements the trait once for its private-state type:

```rust
use compact_runtime::{NoWitnesses, WitnessContext};

#[derive(Clone, Default)]
struct MyPrivateState {
    sk: [u8; 32],
}

struct MyWitnesses;

impl Witnesses<MyPrivateState> for MyWitnesses {
    fn private_secret_key<'a>(
        &self,
        ctx: &WitnessContext<Ledger<'a>, MyPrivateState>,
    ) -> (MyPrivateState, [u8; 32]) {
        // Read sk from somewhere (key store, env, RNG, …).
        (ctx.private_state.clone(), ctx.private_state.sk)
    }
}
```

If the contract declares zero witnesses, the codegen uses
`compact_runtime::NoWitnesses` as the default bound and you can
write `Contract::<_, NoWitnesses>::new(NoWitnesses)`.

### 5. Run the constructor and circuits

```rust
use compact_runtime::*;

// 1. Empty state + private-state seed.
let empty_state = empty_charged_state();
let priv0 = MyPrivateState { sk: [42u8; 32] };

let ctor_ctx = ConstructorContext {
    initial_private_state: priv0,
    empty_zswap_local_state: ZswapLocalState::default(),
    cost_model: initial_cost_model(),
    gas_limit: None,
};

// 2. Run the constructor.
let contract = Contract::new(MyWitnesses);
let ctor = contract
    .initial_state(ctor_ctx, /* v: Fr = */ Fr::from(7u64))
    .expect("constructor failed");

// 3. Drive a circuit (chained — each result carries forward the new
//    state inside `context`).
let circ_ctx = CircuitContext::new(
    ctor.current_contract_state,
    ctor.current_private_state,
);

let after_get = contract.get(circ_ctx).expect("get failed");
// after_get.result is `Maybe<Fr>` — None if state is unset, Some(value) otherwise.

// 4. Read public state through the Ledger view.
let ledger_view = ledger(&after_get.context.current_query_context.state);
let v = ledger_view.value().expect("ledger read failed");
assert_eq!(v, Fr::from(7u64));
```

A complete worked example with every circuit (`set`, `get`, `clear`)
lives in [`tests-e2e-rust/src/tiny.rs`](../tests-e2e-rust/src/tiny.rs) —
it is the byte-parity reference for the TypeScript backend and is
the best fully-runnable end-to-end snippet to copy from.

## Feature support matrix

The list below is sourced from the upstream parity gap report
[`docs/superpowers/research/2026-06-02-upstream-parity-gap-report.md`](../docs/superpowers/research/2026-06-02-upstream-parity-gap-report.md).
If your contract uses a feature in the **Not yet supported** column,
`compactc --rust` will fail with a clear `unsupported Compact
construct` error pointing at the offending node. The TS backend is
unaffected — drop the `--rust` flag and you get the same TS output
you'd get from upstream `compactc`.

### Primitive types

| Feature | `--rust` | Notes |
|---|---|---|
| `Field` | Supported | Lowers to `compact_runtime::Fr`. |
| `Boolean` | Supported | Lowers to `bool`. |
| `Bytes<N>` | Supported | Lowers to `[u8; N]`. |
| `Vector<N, T>` | Supported | Lowers to `[T; N]`. |
| `Uint<8>` … `Uint<128>` | Supported | Lowers to `u8` … `u128`. |
| `Uint<L..U>` (bounded range) | Partial | Power-of-two byte widths lower correctly; arbitrary `L..U` ranges are accepted at the structural level but no generic-struct path exists. |
| `Opaque<"string">` / `Opaque<T>` | Supported | Lowers via `compact_runtime::std_lib::OpaqueString`. |
| `Maybe<T>` | Supported | `compact_runtime::Maybe<T>` + `some<T>(v)` / `none<T>()`. |

### Compact standard-library ADTs

| ADT | `--rust` | Notes |
|---|---|---|
| `Counter` | Supported | |
| `Cell<T>` | Supported | |
| `Map<K, V>` (including nested `Map<K, Map<…>>`) | Supported | `.insert(k, v)`, `.lookup(k)`, `.member(k)` all emit. |
| `Set<T>` | Supported | `.insert(x)`, `.member(x)` emit. `.size()` / `.isEmpty()` not exercised. |
| `MerkleTree<H, T>` | Supported | |
| `HistoricMerkleTree<H, T>` | Supported | `.checkRoot(r)` works. `.insertIndexDefault` not yet. |
| `List<T>` | Supported | `pushFront` / `popFront` / `head` / `length` / `isEmpty`. |

### User-defined types

| Feature | `--rust` | Notes |
|---|---|---|
| `struct S { … }` (exported and non-exported) | Supported | `Aligned` / `FieldRepr` / `FromFieldRepr` impls are emitted automatically. |
| Generic structs `struct S<a, #n> { … }` | Not yet | Compiler errors out. |
| `enum E { variantA, variantB }` | Supported | `default<E>` lowers to `E::variantA`. |
| `type Alias = …` (transparent) | Supported | |
| `type Alias = …` (nominal / newtype) | Supported | |
| Chained aliases | Untested but expected to work | |

### Circuits

| Feature | `--rust` | Notes |
|---|---|---|
| `export circuit f(…): T { … }` | Supported | |
| `circuit f(…): T { … }` (non-exported, called locally) | Supported | |
| `circuit f(…): T { … }` (cross-circuit call) | Supported | |
| `pure circuit f(…): T { … }` (modifier) | Supported in practice | Emitter does not branch on the `pure` keyword today, so `pure` circuits compile but are treated the same as non-pure. |
| Generic circuits `circuit f<#N>(…)` | Not yet | |
| Constructor with parameters | Supported | |
| Implicit (zero-arg) constructor | Supported | |

### Witnesses

| Feature | `--rust` | Notes |
|---|---|---|
| Top-level `witness f(…): T;` | Supported | Generates one trait method per declaration. |
| Module-local `witness` | Supported | Inlined into the importing scope by `expand-modules-and-types` (analysis-passes.ss:43) — emitted as a top-level trait method. |

### Control flow

| Feature | `--rust` | Notes |
|---|---|---|
| `if`/`else` (statement and expression form) | Supported | |
| Ternary `cond ? a : b` | Supported | |
| `for (const i of L..U) { … }` (range loop) | Supported | |
| `for (const x of iterable) { … }` (element loop) | Supported | Literal iterables with unused loop variable cover the common case. |
| `fold((acc, x) => …, init, vec)` | Supported (basic) | Closed-over variables and side-effect-free accumulators work. |
| `map(fn, arr)` | Not yet | |
| Bare lambdas in expression position `(x) => …` | Not yet | |
| Comma-sequenced expressions | Supported | |
| `assert(cond, "msg")` | Supported | Lowers to `compact_assert!`. |

### Modules

The Compact frontend pass `expand-modules-and-types` (at [`compiler/analysis-passes.ss:43`](../compiler/analysis-passes.ss), `Lpreexpand → Lexpanded`) inlines a module's exported declarations into the importing program's namespace **before** the IR reaches the Rust codegen. The `Lexpanded` language definition ([`compiler/langs.ss:529-535`](../compiler/langs.ss)) strips the `(module ...)` and `(import ...)` nonterminals entirely, so the Rust emitter never sees a module boundary — fields, circuits, and witnesses declared inside `module M { ... }` are emitted as flat top-level declarations at the same slot indices and method positions as if they had been declared at the program root. Mirrors how `expand-modules-and-types` also monomorphises generics pre-codegen (see [Iter 11](../tests-e2e-rust/README.md#generics-coverage)).

| Feature | `--rust` | Notes |
|---|---|---|
| `module M { … }` / `import M` | Supported | Inlined pre-Rust-IR. Re-export via `export { … }` brings the module's symbols into the contract's public surface. Covered by [`module_fixture.compact`](../examples/module_fixture.compact) + byte-parity test. |
| `import M prefix P_` | Supported | Same desugaring path — the prefix is applied during inlining; the codegen sees the prefixed names as flat top-level declarations. No dedicated fixture (the codegen path is identical to the flat-import case). |
| Nested modules | Supported | Recursively inlined by the same pass. No dedicated fixture (the codegen path is identical). |
| `export { … }` block | Supported | Standalone `export` blocks at the top level and inside modules both work — they're processed by the same frontend pass that resolves module exports. |

| Feature | `--rust` | Notes |
|---|---|---|
| `persistentHash` / `persistentCommit` / `transientHash` / `transientCommit` | Supported | |
| `hashToCurve` | Supported | Routed through `compact_runtime::hash_to_curve`. |
| Jubjub / EC primitives (`ec_add`, `ec_mul`, …) | Supported | |
| Zswap witnesses | Supported | |
| `keccak256` / `sha256` | Mapped (untested) | The runtime exposes the symbols but the byte-parity tests do not exercise them. |

| `disclose(x)` | Supported | |

### Casts and miscellaneous

| Feature | `--rust` | Notes |
|---|---|---|
| `as Uint<…>` / `as Bytes<…>` / `as Field` | Supported (trivial cases) | Explicit-cast-as-expression generally untested. |
| `Bytes[1, 2, 3, …]` literal | Supported | |
| `[a, b, c]` array literal (homogeneous) | Supported | |
| `[a, b, c]` array literal (mixed type) | Not yet | |
| Compound assignment (`+=`) | Supported | |
| Indexed read `v[i]` | Supported | |
| `contract C {}` declarations | Not yet | |

### What happens when you hit a gap

The codegen never emits broken Rust. When it encounters a construct
it doesn't support, it produces a single error message of the form:

```
compactc --rust: unsupported Compact construct (X): <details>
```

where `X` names the offending node kind (e.g. `module`, `generic-circuit`,
`map-call`). The TypeScript backend continues to work in parallel —
this error is fatal only for the Rust target. If you need a feature
that's not on the list, you can:

1. Drop `--rust` and use the TypeScript output instead.
2. Open an issue with the construct name from the error message and
   a minimal repro. The parity gap report tracks the planned
   iteration order.

## Troubleshooting

### `compactc --rust: unsupported Compact construct (X): …`

The compiler hit a construct that the Rust backend does not handle
yet. Look up `X` in the [Feature support matrix](#feature-support-matrix).
If it's listed as **Not yet**, the gap is known. If it's listed as
**Supported**, please file an issue with the contract.

### `compact-runtime version mismatch`

Caused by linking your contract against a `compact-runtime` version
that differs from the one `compactc` was built against. The
generated `lib.rs` opens with:

```rust
compact_runtime::check_runtime_version!("0.16.100");
```

That literal is the version `compactc` expected. Compare against the
`compact-runtime` version in your `Cargo.lock`:

```bash
cargo tree -p compact-runtime
```

Fix by either:
- Bumping `compact-runtime` in your `Cargo.toml` to match the literal,
  or
- Re-running `compactc` with a matching toolchain.

The literal is regenerated every time you re-run `compactc --rust`,
so the safest workflow is to always re-emit after a compiler bump.

### Byte-parity assertion failures

Most users will not hit this — byte-parity is a test concern. If
you are integrating with a TS client that reads the same on-chain
`StateValue` your Rust contract emits, and you see a mismatch, the
codegen has a bug. The byte-parity tests under
[`tests-e2e-rust`](../tests-e2e-rust) exercise every supported
construct against captured TS reference state; please open an issue
with the smallest contract that reproduces the divergence.

### `the trait bound X: From<…> is not satisfied` at call sites

Compact's runtime makes heavy use of `AlignedValue` for serialising
data into the VM. Generated code calls `AlignedValue::from(x)` on
user types and on most primitives. If you introduced a custom Rust
type (e.g. a wrapper around `Fr`) and pass it to a circuit that
expects `Fr`, you'll get a `From` not implemented error.

Rust's orphan rules forbid implementing third-party traits on
third-party types from your downstream crate, so the fix is:
- Convert at the call boundary (`my_wrapper.0` to extract the inner
  `Fr`), or
- Define the Compact-side type as the wrapper, so the codegen emits
  the trait impls for you via `H5–H7`-style struct emission.

The helpers under `compact_runtime::std_lib` (`bytes_from_field_repr`,
`vec_u8_from_field_repr`, `array_from_field_repr`) exist to make
the codegen's deserialiser path orphan-safe; user code can call them
too if you find yourself fighting the type system from inside a
witness implementation.

### `not all trait items implemented: \`<witness>\``

Every Compact `witness f(…): T;` declaration is one method in the
generated `Witnesses<PS>` trait. You must implement all of them on
one type. The compiler error from rustc points at the missing method
name. Two common gotchas:
- The method name uses **snake_case with `_` instead of `$`** — Compact
  `private$secret_key` becomes Rust `private_secret_key`.
- The method signature is `(&self, ctx: &WitnessContext<Ledger<'a>, PS>) -> (PS, T)`,
  not just `-> T`. The first tuple element is the updated private
  state.

## Versioning

`compact-runtime` and `compactc` share a synchronised version string.
The runtime crate exports `COMPACT_RUNTIME_VERSION` (currently
`0.16.100`), and `compactc --runtime-version` prints the same string.

```bash
compactc --runtime-version
# → 0.16.100
```

When you upgrade the compiler:

1. Re-run `compactc --rust` on every `.compact` source. The emitted
   `check_runtime_version!` literal moves to the new version.
2. Bump the `compact-runtime` dependency in your `Cargo.toml`
   accordingly.

The version pin is **exact**, not semver-ranged: a `0.16.100`
contract will refuse to link against a `0.16.101` runtime, on the
assumption that any prelude change might shift the ABI of generated
code. If we move to a looser pin in future, this section will be
updated.
